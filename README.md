# apkc

apkc is a bare-bones Android app build system. It lets you build simple native Android apps in APK or AAB format without the entire Android IDE. It does a subset of what the Android Gradle plugin does — fast builds with low resource usage. apkc is suitable for learning, prototyping, and small projects.

## Features

- Base project has zero dependencies. APK is <20 KB.
- Builds base project in under 3 seconds.
- Supports incremental builds (`-ic`) with time-based or hash-based change detection (`-h`).
- Supports Java 8 source encoding options (`--utf8`, `--gbk`, `--utf8bom`).
- Supports external JAR libraries and native shared libraries.
- Supports Kotlin source files (`.kt`).
- Supports raw assets.
- Code completion can be set up in VSCode or other editors.
- Cross-platform: macOS, Linux, Windows, FreeBSD, OpenBSD, NetBSD.
- Outputs APK or AAB (Android App Bundle) with production signing support.

## Prerequisites

Install JDK and Android SDK and set the following environment variables:

- `ANDROID_HOME` — path to the Android SDK (e.g. `C:\Users\you\AppData\Local\Android\Sdk`)
- `JAVA_HOME` — path to the JDK (e.g. `C:\Program Files\Java\jdk-17`)

apkc auto-detects these on first run and will locate `aapt2`, `d8`, `zipalign`, `apksigner`, `adb`, and `android.jar` automatically.

## Installation

Download the prebuilt binary for your OS and architecture from the [GitHub Releases](https://github.com/ajinasokan/apkc/releases) page, then add it to your `PATH`.

Alternatively, build from source:

```shell
git clone https://github.com/ajinasokan/apkc.git
cd apkc
go build -o apkc .
```

## Verifying the environment

Run `doctor` to check that all required SDK paths are accessible:

```shell
$ apkc doctor

[doctor] java /usr/local/opt/openjdk@11/
[doctor] javac /usr/local/opt/openjdk@11/bin/javac
[doctor] jarsigner /usr/local/opt/openjdk@11/bin/jarsigner
[doctor] sdk /Users/you/Library/Android/Sdk
[doctor] aapt /Users/you/Library/Android/Sdk/build-tools/34.0.0/aapt
[doctor] aapt2 /Users/you/Library/Android/Sdk/build-tools/34.0.0/aapt2

[doctor] zipalign /Users/you/Library/Android/Sdk/build-tools/34.0.0/zipalign

```

## Creating a project

```shell
$ apkc create myapp

Package name (com.myapp): com.ajinasokan.myapp
App name (My App): Hello World
```

This creates the following structure:

```
myapp/
├── src/com/myapp/
│   └── MainActivity.java
├── AndroidManifest.xml
├── res/
│   ├── layout/main.xml
│   └── values/styles.xml
├── lib/                    # native .so libraries
│   ├── arm64-v8a/
│   ├── armeabi-v7a/
│   ├── x86/
│   └── x86_64/
└── build/                  # generated by apkc
```

## Building

```shell
$ cd myapp
$ apkc build

[build] compiling resources
[build] bundling resources
[build] compiling java files
[build] bundling classes and jars
[build] signing apk
[build] running zipalign
[apkc] finished in 1.580135483s
```

Output is written to `build/app.apk` (or `build/app.aab` with `--aab`).

### Incremental build (`-ic`)

Skip unchanged source files for faster rebuilds:

```shell
$ apkc build -ic
```

Two change-detection modes are available:

| Mode | Flag | How it works |
|------|------|--------------|
| Time-based | `-ic` | Compares file modification timestamps |
| Hash-based | `-ic -h` | Compares SHA-256 hashes of all source files |

Hash mode is more accurate (survives clock skew or file copying) but slower.

### Encoding

By default, apkc uses UTF-8 for Java source encoding. You can override this:

```shell
$ apkc build --utf8    # UTF-8 (default)
$ apkc build --gbk     # GBK (for legacy Chinese source files)
$ apkc build --utf8bom # UTF-8 with BOM
```

Only one encoding flag may be specified at a time.

### Custom keystore

```shell
$ apkc build \
    --keystore "/path/to/keystore.jks" \
    --storepass "password" \
    --keyalias "myalias" \
    --sigalg "SHA256withRSA"
```

If `--keystore` is not specified, apkc uses the debug keystore at the default Android location.

### Building AAB

```shell
$ apkc build --aab
```

Output is written to `build/app.aab`. AAB is required for publishing to the Google Play Store.

## Running on a device

```shell
$ apkc run
```

This builds the APK, installs it on the connected device (prompts for selection if multiple devices are found), launches the main activity, and streams `logcat` output for the running app.

## Other commands

```shell
apkc clean   # delete the build/ directory
```

## Project layout

| Path | Description |
|------|-------------|
| `src/` | Java and Kotlin source files (subdirectories follow package structure) |
| `res/` | Android resources (layouts, values, drawables, etc.) |
| `AndroidManifest.xml` | App manifest |
| `lib/<arch>/` | Native `.so` libraries per architecture (`arm64-v8a`, `armeabi-v7a`, `x86`, `x86_64`) |
| `jar/` | External JAR library files |
| `assets/` | Raw asset files |
| `build/` | Output directory (generated; safe to delete with `clean`) |

## Code completion in VSCode

1. Install the [Java extension](https://marketplace.visualstudio.com/items?itemName=redhat.java).
2. Find the `android.jar` path from the `apkc doctor` output.
3. Add it to `referencedLibraries` in `.vscode/settings.json`:

```json
{
    "java.project.referencedLibraries": [
        "/path/to/Android/Sdk/platforms/android-34/android.jar"
    ]
}
```

External JAR dependencies in the `jar/` directory are automatically included on the classpath during builds.
